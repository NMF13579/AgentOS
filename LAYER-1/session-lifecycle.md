# Session lifecycle — жизненный цикл сессии и контекста

Единое описание триггеров «старт / конец задачи / сохранить / аудит / фичи», чтобы не дублировать их в файлах IDE.

---

## Старт каждой новой сессии (onSessionStart)

Выполнять автоматически при открытии контекста проекта:

1. Прочитать `LAYER-3/STATE.md` (если файл есть) — домены Project / Session / Task, guards.
2. Прочитать `HANDOFF.md` и `LAYER-3/project-status.md`.
3. Прочитать `LAYER-3/lessons.md`, если файл существует.
4. Если одна и та же задача снова выглядит незакрытой — предупредить владельца:  
   «⚠️ Задача [название] идёт уже несколько сессий. Разбить на части или продолжить?»
5. Кратко (3–5 строк): текущее состояние, что делали в прошлый раз, 1–3 варианта следующего шага.
6. Дождаться выбора владельца — **не писать код**, пока владелец явно не согласовал дальнейшие действия (см. также `LAYER-1/workflow.md`, «Молчание ≠ согласие»).

---

## После каждой завершённой задачи (onTaskComplete)

1. Обновить `HANDOFF.md` — что сделано, где остановились, что дальше.
2. Обновить `LAYER-3/project-status.md`.
3. При необходимости обновить профильные файлы памяти (`LAYER-3/deferred-decisions.md`, `LAYER-3/fixes.md`, `LAYER-3/security.md` и т.д.) — по смыслу изменений.
4. Сообщить владельцу: «Контекст сохранён ✅».

---

## Напоминание о сохранении контекста (onSaveReminder)

Напомнить («⚠️ Контекст не сохранялся. Сохранить сейчас?»), если выполняется **любое** из условий:

- с момента последнего обновления `HANDOFF.md` сделано **10+** правок файлов, **или**
- выполнен план из **3+** шагов без обновления `HANDOFF.md`.

---

## Запрос аудита или «здоровья» проекта (onAuditRequest)

Триггеры вроде: «проверь проект», «аудит», «здоровье».

1. Открыть и следовать протоколу в `LAYER-1/audit.md`.
2. Для короткой проверки — раздел **«Чеклист аудита»** в том же файле (включая HEALTH-SCORE).

---

## Запрос идей («что добавить», «что дальше», после закрытия фазы) (onFeatureRequest)

1. Прочитать `LAYER-1/feature-radar.md` и `LAYER-3/features.md`.
2. Предложить владельцу **три** приоритетных варианта с кратким обоснованием.

---

## Эквиваленты «субагентов» (только документация)

В средах без вызова встроенных субагентов использовать те же триггеры через файлы:

| Триггер / намерение | Действие |
|---------------------|----------|
| «проверь проект», «аудит», «здоровье» | Запустить протокол `LAYER-1/audit.md`; чеклист и HEALTH-SCORE — в разделе **«Чеклист аудита»** того же файла |
| «что добавить», «что дальше», после закрытия фазы | `LAYER-1/feature-radar.md` + `LAYER-3/features.md`, топ-3 |
| 10+ правок или план 3+ шагов без сохранения | Напоминание по разделу **Напоминание о сохранении** выше |

Специализированные субагенты (audit-agent, feature-advisor-agent, session-guard-agent) в **Claude Code** вызываются только там; в остальных средах — только соответствующие документы и таблица выше.
