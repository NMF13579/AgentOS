# ROLLBACK-PROTOCOL-POST-DEPLOY — откат после неудачного деплоя

> Этот протокол — для ситуации «задеплоили → всё сломалось в проде».
> Для откатов в разработке используй `LAYER-1/error-handling.md → ROLLBACK-PROTOCOL`.

---

## Принцип: сначала восстановить прод, потом разбираться

Не чинить причину под давлением — сначала вернуть рабочую версию пользователям.

---

## Шаг 0 — Оценка критичности (1 минута)

Агент задаёт ОДИН вопрос:

> «Что сейчас происходит в продакшене?
> А) Сайт/приложение недоступно полностью
> Б) Одна функция сломана, остальное работает
> В) Данные выглядят неправильно или частично потеряны
> Г) Медленно работает / таймауты
> Д) Другое — опиши»

**При ответе А или В — немедленный откат без обсуждения.**
**При Б, Г, Д — уточнить масштаб, потом решать.**

---

## Шаг 1 — Быстрый откат деплоя (требует подтверждения владельца)

### Vercel
```
# В интерфейсе Vercel:
# Deployments → найти предыдущий рабочий деплой → Promote to Production
# Время: ~30 секунд
```

### Netlify
```
# Deploys → найти предыдущий рабочий деплой → Publish deploy
# Время: ~1 минута
```

### Timeweb / VPS (ручной деплой через git)
```bash
# Откат к предыдущему коммиту
git log --oneline -5          # найти SHA рабочей версии
git revert HEAD               # если последний коммит = последний деплой
# или
git reset --hard <SHA>        # к конкретному коммиту (осторожно: деструктивно)
git push origin main --force  # только если нет других разработчиков
```
> ⚠️ `git push --force` требует явного подтверждения владельца.
> Агент не выполняет его без слов «я понимаю последствия, выполняй».

### Docker / контейнеры
```bash
# Откатить тег образа
docker pull myapp:previous-tag
docker stop myapp-container
docker run -d --name myapp-container myapp:previous-tag
```

## Шаг 2 — Проверка что откат сработал

Агент проверяет и сообщает владельцу:

```
Откат выполнен.

Проверь вручную:
[ ] Главная страница открывается
[ ] Авторизация работает
[ ] [ключевая функция проекта] работает

Что видишь?
```

---

## Шаг 3 — Если откат не помог (данные или БД)

> Только при варианте В (данные неправильные).
> ⚠️ Этот шаг необратим без бэкапа. Требует явного подтверждения.

### Supabase
```
# Supabase Dashboard → Project Settings → Backups
# Выбрать последний бэкап ДО проблемного деплоя
# Восстановить (Point-in-time recovery если доступно)
```

**Для медицинских проектов:**
- Перед восстановлением бэкапа зафиксировать в `LAYER-3/fixes.md` что именно потеряно
- Уведомить ответственного врача / администратора клиники
- После восстановления — проверить целостность данных пациентов выборочно

---

## Шаг 4 — Зафиксировать причину и защититься

После восстановления агент записывает в `LAYER-3/fixes.md`:

```
[Дата] — Post-deploy rollback
Деплой: [SHA или версия]
Симптом: [что сломалось]
Откат: [каким методом]
Прод восстановлен: [да / нет]
Причина поломки: [что стало причиной]
Защита: [что добавить чтобы не повторилось]
```

---

## Что добавить чтобы не повторялось

| Проблема | Защита |
|---------|--------|
| Деплой сломал прод | Staging-окружение перед проддеплоем |
| Данные повреждены | Автобэкап перед каждой миграцией БД |
| Нет точки отката | Git-тег на каждый прод-деплой: `git tag prod-YYYYMMDD` |
| Не заметили сразу | Health-check endpoint + мониторинг (UptimeRobot бесплатно) |
