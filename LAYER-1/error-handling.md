# ERROR-TYPES — Классификация ошибок

> Когда что-то идёт не так — агент называет тип ошибки вслух.
> Это помогает быстро понять причину и выбрать правильное действие.
> Не читать подряд — использовать как справочник при возникновении проблемы.

---

## Типы ошибок

### 🔵 PROCESS ERROR — пропущен этап

**Что это:** агент или владелец пропустили обязательный шаг протокола.

**Примеры:**
- Код написан без подтверждённого плана
- Задача закрыта без прохождения TASK-REVIEW-PROTOCOL
- Новая зависимость добавлена без согласования

**Действие:**
> «Ошибка: PROCESS ERROR. Пропущен шаг: [что именно].
> Возвращаемся и выполняем его сейчас.»

---

### 🟡 LOGIC ERROR — неправильное решение

**Что это:** техническое решение не работает или решает не ту проблему.

**Примеры:**
- Код написан правильно, но делает не то что нужно
- Выбран неподходящий подход к архитектуре
- Запрос к базе возвращает неверные данные

**Действие:**
> «Ошибка: LOGIC ERROR. Решение не соответствует цели задачи.
> Запускаем RETRY RULE. Попытка [1/2]: [новый подход].»

---

### 🟠 CONTEXT ERROR — потеря контекста

**Что это:** агент не знает где мы находимся в проекте или противоречит прошлым решениям.

**Примеры:**
- Агент предлагает то что уже было отклонено
- Агент не знает какая задача активна
- Агент повторяет вопросы на которые уже отвечали

**Действие:**
> «Ошибка: CONTEXT ERROR. Восстанавливаю контекст.»
> → Запустить CONTEXT-LOSS-RECOVERY.md

---

### 🔴 SCOPE ERROR — разрастание задачи

**Что это:** работа выходит за рамки текущей задачи.

**Примеры:**
- В процессе выполнения TASK-003 начали менять код из TASK-001
- Владелец добавляет новые требования в середине задачи
- Агент «улучшает» то что не просили

**Действие:**
> «Ошибка: SCOPE ERROR. Это выходит за рамки TASK-XXX.»
> → Запустить LAYER-1/scope-guard.md

---

### ⚫ BLOCK ERROR — внешний блокер

**Что это:** задача не может быть выполнена из-за внешней зависимости.

**Примеры:**
- Нужен API-ключ которого нет
- Зависит от задачи которая ещё не закрыта
- Нужно решение владельца прежде чем двигаться дальше

**Действие:**
> «Ошибка: BLOCK ERROR. Задача заблокирована: [причина].
> Статус задачи: blocked. Что нужно от тебя: [конкретный вопрос].»
> *(Статус задачи ≠ MODE агента. MODE остаётся текущим.)*

---

### 🟣 SEMANTIC ERROR — сделано не то что просили

**Что это:** код работает, тесты проходят, но результат не соответствует
намерению владельца — агент неправильно понял задачу.

**Отличие от LOGIC ERROR:** LOGIC — техническая ошибка в реализации.
SEMANTIC — правильная реализация неправильно понятой задачи.

**Примеры:**
- Агент сделал фильтрацию по дате, владелец хотел по статусу
- Агент добавил кнопку «Сохранить», владелец хотел автосохранение
- Агент создал форму входа, владелец имел в виду форму регистрации

**Триггер:** владелец говорит «это не то», «не так», «я имел в виду другое»,
«ты неправильно понял»

**Действие:**
> «Ошибка: SEMANTIC ERROR. Я неправильно понял задачу.
> Вот как я понял: [моё понимание].
> Поправь меня: что именно должно быть?»

После уточнения — **не RETRY**, а сразу **MODE = PLANNING**:
заново составить план с правильным пониманием и показать владельцу.

**Что НЕ делать:**
- Не пытаться «угадать» и исправить молча
- Не применять RETRY RULE (это не техническая ошибка)
- Не начинать переделку без нового подтверждённого плана

## Реактивный алгоритм после ошибки (дерево решений)

`Self-Verification = до выполнения.`  
`Error-Handling = после падения.`

Алгоритм:

1. Ошибка зафиксирована (компилятор, рантайм, API, регрессия).
2. Классифицируй тип ошибки из разделов выше.
3. Попытайся авто-исправление максимум в 2 итерации.
4. Если не удалось — переходи к разделу **«Процедура отката»** в конце этого файла.

Короткая форма:

```text
Ошибка → Классификация → Авто-исправление (1/2, 2/2) → Неудача → Процедура отката (ниже)
```

Для сценариев «неустранимая ошибка» и «повторяющийся сбой» откат обязателен.

## Как агент сообщает об ошибке

Единый формат:

```
[ТИП ОШИБКИ] — [одна строка что произошло]
Причина: [почему это произошло]
Действие: [что делаем дальше]
```

Пример:
```
SCOPE ERROR — запрос на добавление фильтрации в ходе TASK-005
Причина: фильтрация не входит в критерии готовности задачи
Действие: добавляю в Backlog ROADMAP.md, продолжаем TASK-005
```

## Правило для владельца

Если агент называет тип ошибки — это не упрёк, это диагноз.
Каждый тип имеет чёткое действие. Спорить с типом не нужно — нужно выполнить действие.

## Когда сразу запускать откат

Запускай протокол отката (раздел **«Процедура отката»** ниже), если:

- ошибка не устраняется за 2 итерации;
- сбой повторяется после «исправления»;
- есть риск для безопасности, данных или стабильности ядра;
- обнаружен выход за scope с поломкой текущего функционала.

## Процедура отката

> Безопасный возврат проекта в стабильное состояние через GUI Cursor/Claude Code.  
> *(Перенесено из бывшего `ROLLBACK.md`.)*

### Триггеры активации

Запускай протокол сразу, если произошёл хотя бы один пункт:

- Критическая ошибка генерации, из-за которой результат нельзя безопасно продолжать.
- Нарушение безопасности (утечка ПДн, попытка обхода ограничений, опасная команда).
- Поломка существующего функционала после последних изменений.
- Агент вышел за границы задачи (scope) и это затронуло рабочую часть проекта.

### Пошаговый алгоритм (GUI-ориентированный)

#### 1) Остановка агента

- В Cursor нажми **Stop** у текущего запуска.
- Альтернатива: команда `/cancel` в чате.
- Подтверди, что новых правок больше не выполняется.

#### 2) Фиксация текущего состояния

- Открой Source Control в IDE и сохрани список изменённых файлов.
- Если нужно, экспортируй текущий diff через GUI или скопируй список изменений в заметку инцидента.
- При наличии Git создай безопасную «точку снимка» через интерфейс коммита (без деплоя).

#### 3) Возврат к последней стабильной точке

- Встроенный Git/UI Cursor: выбери последний стабильный коммит/изменение.
- Выполни откат через интерфейс (`Discard/Revert` для нужных файлов или возврат к стабильному коммиту).
- Проверь, что критичный сценарий снова работает.

#### 4) Логирование инцидента

После отката обязательно добавь запись в `audit-log.md` по шаблону:

```md
## [YYYY-MM-DD HH:MM] Откат изменений
- Причина: [что сломалось / какой риск обнаружен]
- Область: [файлы / модуль / сценарий]
- Что откатили: [какую точку и почему выбрали]
- Итог: [что восстановлено]
- Решение на будущее: [как не повторить инцидент]
```

#### 5) Переформулировка задачи и перезапуск

- Сформулируй новую, суженную задачу в 1–3 пунктах.
- Явно укажи границы scope и критерий готовности.
- Запусти агента заново только после подтверждения владельца.

### Чего НЕ делать

- Не удалять файлы вручную «вслепую», чтобы быстро скрыть проблему.
- Не игнорировать логи и журнал инцидента.
- Не продолжать работу без отката, если стабильность уже нарушена.

## Когда откатывать vs исправлять на месте

| Действие | Исправлять на месте | Откатывать (протокол выше) |
|----------|---------------------|----------------------------|
| **Итерации** | 1–2 попытки по дереву error-handling, есть ясный минимальный фикс | Уже 2 неудачные итерации или сбой повторяется после «фикса» |
| **Риск** | Нет угрозы данным, секретам, прод-стабильности | Есть риск для безопасности, ПДн, ядра продукта |
| **Scope** | Ошибка внутри согласованной задачи | Выход за scope + сломан существующий функционал |
| **Понимание** | Воспроизводимо, причина ясна | Хаос в диффе, неясно что безопасно оставить |

**Правило большого пальца:** пока можно честно сказать «один точечный патч устранит это без побочных эффектов» — чиним на месте. Если нет уверенности или патч разрастается — фиксируем состояние и откатываемся к последней зелёной точке, затем сужаем задачу.
