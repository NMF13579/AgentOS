# DECISION-GUIDE — Справочник развилок проекта

> Этот файл агент использует когда нужно объяснить владельцу выбор между технологиями или подходами.
> Каждая развилка — простое объяснение без жаргона + конкретная рекомендация.

---

## Ключевые термины (самодостаточно)

- **Scope-guard** — правило границ задачи: если запрос выходит за согласованный объём, агент останавливается, показывает риск и просит явный выбор действия.
- **Risk-tag** — тег риска задачи (`[РИСК: HIGH]`, `[РИСК: MEDIUM]`, `[РИСК: LOW]`), который определяет глубину проверки перед завершением.
- **Self-verification** — внутренняя проверка агента до выполнения/коммита: корректность логики, безопасность, соответствие границам задачи.

---

## Матрица принятия решений

| Ситуация | Вопрос | Критерий выбора | Следующий шаг |
|---|---|---|---|
| Запрос расширяет текущую задачу | Это критично для текущего результата или можно вынести отдельно? | Если без изменения нельзя завершить текущую цель — фиксируем расширение; иначе переносим | Остановиться, применить `scope-guard`, зафиксировать решение |
| Работа с медицинскими данными пациента | Данные идентифицируют пациента или могут повлиять на клиническое решение? | Если да — автоматически `[РИСК: HIGH]` | Включить высокий уровень проверки и правовой контур 152-ФЗ |
| Выбор технологии для MVP | Решение ускоряет запуск без блокера масштаба? | Если да — выбираем более простой путь с обратимой архитектурой | Зафиксировать выбор и критерий пересмотра |
| Ошибка в процессе выполнения | Ошибка устранима за 1–2 попытки без риска регрессии? | Если нет — не чинить бесконечно | Активировать безопасный откат и перезапуск |

## Как пользоваться этим файлом

Когда в разговоре с владельцем возникает вопрос типа «а какая разница?» или «что лучше?» —
агент находит нужную развилку здесь и объясняет её своими словами, опираясь на контекст проекта.

**Агент никогда не называет оба варианта равнозначными** — он всегда рекомендует конкретный вариант
для данного проекта и объясняет почему.

Нужные переходы в другие документы:

- В `workflow.md` — когда после выбора нужно перейти к пошаговому выполнению.
- В `scope-guard.md` — когда решение затрагивает границы текущей задачи.
- В `task-protocol.md` — когда нужно выбрать и зафиксировать уровень риска проверки.

## Развёрнутые примеры принятия решений

### Пример 1 — Пациент загружает фото рецепта

**Ситуация:** пользователь загружает изображение рецепта для разбора текста.  
**Решение:** это `[РИСК: HIGH]`, потому что затрагиваются медицинские данные и возможны ошибки интерпретации.  
**Критерии:** нужен OCR, проверка качества распознавания, валидация критичных полей и human-апрув перед клиническим действием.  
**Следующий шаг:** запуск по `workflow.md` с фиксацией риска в задаче и отдельной проверкой legal/security контуров.

### Пример 2 — Врач меняет назначение дозировки

**Ситуация:** в интерфейсе нужно добавить изменение дозировки по пациенту.  
**Решение:** `[РИСК: HIGH]`, так как ошибка напрямую влияет на безопасность пациента.  
**Критерии:** двойная валидация диапазона, обязательное подтверждение действия, журналирование решения и ответственного врача.  
**Следующий шаг:** сначала спецификация правила дозировки, затем реализация, затем human audit перед выпуском.

### Пример 3 — Пациентский календарь записи

**Ситуация:** требуется улучшить UX выбора слота при записи на приём.  
**Решение:** `[РИСК: MEDIUM]`, так как это бизнес-логика и UX без прямого клинического решения.  
**Критерии:** корректная фильтрация слотов, понятные сообщения, smoke-тест человеком на 2–3 сценариях.  
**Следующий шаг:** идти по стандартному циклу план → подтверждение → реализация → smoke-проверка.

## РАЗВИЛКА 1 — Где хранить данные: Supabase vs Firebase

**Простое объяснение:**

Оба — это облачные базы данных, куда твоё приложение сохраняет информацию.

| | Supabase | Firebase |
|---|---|---|
| **Что это** | База данных как у больших компаний (PostgreSQL), просто с удобным интерфейсом | База данных от Google, специально созданная для мобильных приложений |
| **Кому подходит** | Если у тебя есть таблицы, связи между данными, фильтры | Если нужно что-то простое и быстрое, особенно чат или реалтайм |
| **Цена** | Бесплатно до ~50 тыс. запросов в день | Бесплатно до определённого лимита, потом дороже |
| **Сложность** | Чуть сложнее на старте, но мощнее | Проще на старте, но сложнее масштабировать |

**Рекомендация агента:** Для большинства MVP — Supabase. Он открытый, можно перенести данные куда угодно, и хорошо работает с Lovable/Bolt.

## РАЗВИЛКА 2 — Как сделать интерфейс: Lovable vs Bolt vs FlutterFlow

**Простое объяснение:**

Все три — это инструменты для создания визуальной части приложения без написания кода вручную.

| | Lovable | Bolt | FlutterFlow |
|---|---|---|
| **Что делает** | Создаёт веб-приложение через чат с ИИ | То же, но быстрее и без ограничений на запросы | Создаёт мобильное приложение через конструктор |
| **Лучше всего для** | Полноценные веб-SaaS, сложные формы, дашборды | Прототипы, эксперименты, быстрые MVP | Мобильное приложение под iOS и Android |
| **Интеграция с Supabase** | Отличная | Хорошая | Хорошая |
| **Где запускается** | В браузере | В браузере | В телефоне |

**Развилка:** Тебе нужно приложение в браузере или в телефоне?
- **Браузер** → Lovable или Bolt
- **Телефон** → FlutterFlow

## РАЗВИЛКА 3 — Как авторизовать пользователей

**Простое объяснение:**

Авторизация — это система «войти / зарегистрироваться» в твоём приложении.

| Вариант | Что это | Когда выбирать |
|---|---|---|

| **Clerk** | Отдельный сервис авторизации. Красивые формы, много настроек | Если нужны сложные роли, организации, детальное управление доступом |

**Рекомендация агента:** Supabase Auth — бесплатно, уже встроено, работает сразу. Clerk — только если Supabase Auth не справляется с требованиями.

## РАЗВИЛКА 6 — Монолит vs Микросервисы

**Простое объяснение:**

Это вопрос о том, как организован код внутри.

**Монолит** — всё в одном месте. Как один большой завод, где все цеха рядом.
- Просто разрабатывать
- Просто деплоить
- Хорошо для MVP

**Микросервисы** — много маленьких отдельных частей. Как город из маленьких магазинчиков.
- Сложнее на старте
- Легче масштабировать каждую часть отдельно
- Имеет смысл только когда пользователей уже десятки тысяч

**Рекомендация агента:** Для MVP — всегда монолит. Переход к микросервисам происходит органически когда проект вырастает, не раньше.

## РАЗВИЛКА 8 — SQL vs NoSQL база данных

**Простое объяснение:**

Это два разных способа хранить данные.

**SQL (таблицы)** — как Excel. Строки и столбцы. Данные связаны между собой.
- Пример: пользователь → его заказы → товары в заказе
- Supabase, PostgreSQL

**NoSQL (документы)** — как куча JSON-файлов. Каждый объект хранит всё сам в себе.
- Пример: каждый пост в соцсети хранит автора, текст, лайки вместе
- Firebase Firestore, MongoDB

**Рекомендация агента:** Для большинства продуктов — SQL (Supabase). NoSQL выбирается только если данные действительно «плоские» и не связанные, или нужен реалтайм-чат.

## РАЗВИЛКА 10 — Один язык/фреймворк для всего vs специализированные

**Простое объяснение:**

Иногда можно использовать один инструмент для фронта и бэка (например Next.js), а иногда лучше разделить.

**Один стек (Next.js / Nuxt):**
- Проще для небольших команд
- Lovable и Bolt создают именно такой стек
- Отлично для MVP

**Разделённый стек (React фронт + отдельный бэкенд):**
- Лучше масштабируется
- Сложнее на старте
- Имеет смысл если фронт и бэк делают разные люди

**Рекомендация агента:** Для MVP и вайб-кодинга — один стек. Разделение — только если есть конкретная причина.

## РАЗВИЛКА 11 — Хранение медицинских данных: своя схема vs FHIR

**Контекст:** ты проектируешь хранилище данных пациентов или медкарт.

| Вариант | Когда выбирать |
|---------|---------------|
| **Своя схема (PostgreSQL + RLS)** | MVP, внутренний инструмент, нет интеграции с внешними МИС, команда 1–3 человека |
| **FHIR R4 (через Medplum или самостоятельно)** | Нужна интеграция с федеральными системами, планируется обмен с другими МИС, долгосрочный продукт |
| **HL7 v2 (legacy)** | Интеграция с существующей МИС, которая поддерживает только HL7 v2 |

**Рекомендация для РФ:** начинай со своей схемой на MVP, закладывай FHIR-совместимые ID уже с первой итерации (поле `fhir_id` в таблицах patients/encounters). Переход на FHIR потом будет дешевле.

**Триггер для Medical Reviewer:** если выбирается FHIR или интеграция с МИС → обязательный review.

## РАЗВИЛКА 12 — Интеграция с МИС (1С:Медицина, МедИС, БАРС)

**Контекст:** нужно получать или отдавать данные в существующую МИС клиники.

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **REST API МИС** | Просто, если у МИС есть API | Есть не у всех, документация слабая |
| **HL7 FHIR-фасад** | Стандартизированно, переносимо | Нужна разработка фасада |
| **Прямой доступ к БД МИС** | Быстро на MVP | Нарушает изоляцию, риски безопасности, нет поддержки от вендора |
| **Файловый обмен (CSV/XML)** | Работает всегда | Не real-time, ошибки синхронизации |

**Рекомендация:** REST API если есть → FHIR-фасад если нет → файловый обмен как крайний случай. Прямой доступ к БД МИС — только если нет других вариантов и с письменным согласованием ИТ-службы клиники.

## РАЗВИЛКА 13 — Идентификатор пациента: что использовать

**Контекст:** ты проектируешь таблицу пациентов и нужно выбрать первичный идентификатор.

| Идентификатор | Когда подходит | Риски |
|--------------|---------------|-------|
| **Внутренний UUID** | Всегда, как основной PK | Нет привязки к внешним системам |
| **СНИЛС** | Интеграция с ЕГИСЗ, ОМС | Обязателен 152-ФЗ режим хранения, нельзя в URL |
| **Номер полиса ОМС** | Работа со страховыми | Меняется при смене страховой |
| **МИС-ID** | Интеграция с конкретной МИС | Привязка к вендору |

**Рекомендация:** UUID как PK всегда. СНИЛС и ОМС — как дополнительные поля с шифрованием (не в открытом виде). Никогда не использовать СНИЛС или ОМС в URL, логах, публичных полях API.

## РАЗВИЛКА 14 — Хранение согласий пациента (152-ФЗ / 323-ФЗ)

**Контекст:** приложение собирает ПДн пациентов или оказывает медицинские услуги.

| Вариант | Когда подходит |
|---------|---------------|
| **Чекбокс в форме + timestamp в БД** | MVP, нет юридических требований к форме согласия |
| **Отдельная таблица consents (пациент, дата, версия, статус, IP)** | Минимум для 152-ФЗ — рекомендуется всегда |
| **Электронная подпись (КЭП/НЭП)** | Телемедицина, юридически значимые документы |
| **Бумажное согласие + скан в системе** | Стационар, клиника с физическим приёмом |

**Обязательные поля таблицы `consents`:**
`patient_id, consent_type, granted_at, expires_at, revoked_at, version, ip_address, agent_id`

**Рекомендация:** таблица `consents` обязательна при любой работе с ПДн пациентов.
Право пациента на отзыв согласия — реализовать с первой итерации (поле `revoked_at`).
При отзыве — данные не удалять, а блокировать (soft delete + аудит).

## РАЗВИЛКА 15 — Один агент vs мульти-агентный пайплайн

**Контекст:** проект вырос, задачи стали сложнее, хочется ускорить работу.

| Вариант | Когда выбирать |
|---------|---------------|
| **Один агент (текущий режим)** | MVP, проект < 6 месяцев, владелец один, задачи последовательные |
| **Мульти-агент через `.claude/agents/`** | Есть параллельные независимые задачи (фронт + бэк + тесты), проект > 6 месяцев, команда 2+ человек |
| **Специализированные агенты по ролям** | Нужен Medical Reviewer как отдельный агент, отдельный Docs Agent для документации |

**Сигналы что пора в мульти-агент:**
- Одна задача занимает > 3 часов агентского времени
- Часто нужен код + review + docs одновременно
- Появился второй владелец / разработчик

**Сигналы что рано:**
- Задачи строго последовательные
- Владелец не успевает за одним агентом
- Нет чёткого разграничения зон ответственности

## РАЗВИЛКА 16 — Когда переходить с Level 0 на Level 1/2

**Контекст:** проект работает, но непонятно когда «повышать уровень» документации.

| Переход | Триггер |
|---------|---------|
| **Level 0 → Level 1** | Есть реальные пользователи (хотя бы 1), или планируется деплой в прод, или проект длится > 1 месяца |
| **Level 1 → Level 2** | Команда > 1 человека, или критичные данные (медицина, финансы), или нужна интеграция с внешними системами |
| **Остаться на Level 0** | Прототип / проверка гипотезы, срок < 2 недель, пользователей нет |

**Что делать при переходе:**
1. Запустить `LAYER-1/audit-checklist.md` — HEALTH-SCORE текущего состояния
2. Заполнить недостающие файлы `LAYER-2/specs/`
3. Зафиксировать переход в `LAYER-3/project-status.md`

## РАЗВИЛКА 17 — Когда дробить задачу на подзадачи

**Контекст:** задача большая, непонятно всё ли делать сразу.

| Признак | Действие |
|---------|---------|
| Задача занимает > 3 шагов плана | Разбить на 2–3 TASK с отдельными AC |
| Задача затрагивает > 3 файлов одновременно | Разбить: сначала структура, потом логика, потом UI |
| Задача включает «и» — «сделай X и Y» | Разбить по «и» — каждое «и» = отдельный TASK |
| Задача меняет схему БД | Отдельный TASK только для миграции БД перед кодом |
| Первое действие можно проверить отдельно | Разбить — быстрее найдёшь что сломалось |

**Сигнал агента при обнаружении большой задачи:**
```
Эта задача большая — вижу [N] независимых частей.
Предлагаю разбить:
TASK-A: [часть 1] — результат: [что проверяем]
TASK-B: [часть 2] — результат: [что проверяем]

Делаем так или всё одним шагом?
```
