# Document Priority — приоритет источников и ответы о проекте

Репозиторий — единственный источник правды; знания модели — только запасной вариант.

## Разрешение конфликтов (канон)

Полная иерархия: [`shared/priority-order.md`](../shared/priority-order.md).

При конфликте инструкций следовать порядку (1 = высший приоритет):

1. Активный файл задачи в `tasks/TASK-XXX.md`, если есть открытая задача.
2. `project/PROJECT.md` + `HANDOFF.md` + `LAYER-3/project-status.md`.
3. Активный `stages/*/BOOT.md` (поле `priority` во frontmatter).
4. `shared/scope-guard.md` → `shared/agent-contract.md` → документ этапа → `shared/workflow.md`.
5. Тематический файл в `LAYER-1/` или `LAYER-2/`.
6. `CLAUDE.md` и `LAYER-1/system-prompt.md` (для шаблона репозитория — как указатели; каноническая логика в `LAYER-1/`).

Если два источника **одного** уровня противоречат друг другу — **остановиться** и спросить владельца, какой документ главнее для ситуации. **Не** выбирать молча.

## Перед ответом на вопрос о проекте

- Свериться с иерархией выше, затем искать ответ в репозитории.
- Ссылаться на файл явно: «По `LAYER-3/project-status.md`: …»
- При противоречии документов — указать на конфликт, не подменять ответ.
- Если данных нет — сказать, чего не хватает и куда это логично добавить.

## Самопроверка перед ответом с кодом

Перед тем как писать код, мысленно пройти:

- [ ] Прочитан ли релевантный `BOOT.md` для текущего этапа?
- [ ] Проверен ли `LAYER-2/specs/architecture.md` на решения по стеку?
- [ ] Проверен ли `LAYER-3/deferred-decisions.md` на отложенные решения?
- [ ] Согласован ли план с текущей задачей в `HANDOFF.md`?
- [ ] В рамках ли подтверждённого скоупа действие?

Если хотя бы один пункт «нет» — сначала дочитать недостающий документ, затем отвечать.
