# ANTI-PATTERNS — ошибки которые нельзя повторять

> Агент читает этот файл при выборе подхода к решению.
> Каждый пункт — это реальная ошибка которую легко совершить.

---

## Ошибки в процессе

### ❌ Писать код без подтверждённого плана
Код без понимания = переделки. Всегда: план → подтверждение → код.

### ❌ Делать несколько вещей за один промпт
«Добавь авторизацию, исправь баг и переделай дизайн» → хаос.
Одна задача — один промпт.

> ⚠️ Это правило — про пользовательские задачи и вопросы к владельцу.
> Внутренние операции системы (обновление HANDOFF.md, memory-bank, project-status.md)
> под него не попадают — они выполняются как единое завершение задачи, полностью.

### ❌ Менять рабочий код «заодно»
Если задача — добавить кнопку, не трогать логику авторизации «пока открыт файл».

### ❌ Не обновлять HANDOFF.md после сессии
Следующая сессия начнётся с нуля. Потеря прогресса.

### ❌ Двигаться без «всё верно» от владельца
Молчание ≠ согласие. Ждать явного подтверждения.

## 🔄 Процессуальные и коммуникационные анти-паттерны

### 🔴 Постоянные уточнения

**Описание:** условия задачи меняются в середине выполнения без фиксации новой версии задачи.  
**Пример:** «Оставь текущую форму, но добавь ещё два шага, интеграцию и новый отчёт прямо сейчас».  
**Последствия:** агент теряет контекст, срок задачи расползается, растёт риск ошибок.  
**Решение:** каждое изменение фиксировать через процедуру `scope-guard` и подтверждать перед продолжением: [`scope-guard.md`](./scope-guard.md).

### 🟡 Игнорирование скоупа

**Описание:** попытка добавить «маленькую доработку» без пересогласования.  
**Пример:** «Пока правишь фильтр, сразу добавь экспорт в Excel».  
**Последствия:** границы задачи размываются, ломается трассируемость прогресса.  
**Решение:** применять процедуру расширения скоупа с остановкой и выбором действия: [`scope-guard.md`](./scope-guard.md).

### 🟠 Задача без критериев приёмки

**Описание:** задача сформулирована абстрактно и не проверяется объективно.  
**Пример:** «Сделай удобно и современно».  
**Последствия:** агент начинает гадать, результат сложно принять или отклонить.  
**Решение:** всегда указывать измеримый критерий в формате `Успех = [конкретный результат]`: [`task-protocol.md`](./task-protocol.md).

### 🟡 Смешивание ролей

**Описание:** заказчик одновременно пытается управлять архитектурой, реализацией и проверкой в одном сообщении.  
**Пример:** «Я сам допишу половину, а ты сразу закоммить и проверь релиз».  
**Последствия:** конфликт контекста, ответственность размыта, ошибки выявляются поздно.  
**Решение:** разделять поток `Заказчик → Архитектор → Агент` и фиксировать текущую роль на шаг: [`agent-contract.md`](./agent-contract.md).

## Ошибки в коде

### ❌ Хардкодить значения которые могут измениться
```javascript
// Плохо
const API_URL = "https://api.example.com/v1"

// Хорошо
const API_URL = process.env.VITE_API_URL
```

### ❌ Копировать секреты в код
API ключи, пароли, токены — только в `.env` файл. Никогда в код.

### ❌ Удалять данные без проверки
Любой `DELETE` без `WHERE` — катастрофа. Всегда указывать условие.

### ❌ Менять схему базы данных вручную
Только через миграции. Иначе разные окружения рассинхронизируются.

### ❌ Делать всё в одном файле
Один файл > 300 строк — сигнал что нужно разбить.

## Ошибки в общении с владельцем

### ❌ Использовать жаргон без объяснения
«Нужно добавить индекс в БД» → непонятно.
«Нужно ускорить поиск — добавим специальную метку в базу данных» → понятно.

### ❌ Задавать несколько вопросов сразу
Список из 5 вопросов = ступор. Один вопрос за раз.

### ❌ Предлагать варианты без рекомендации
«Можно сделать А или Б» → неудобно.
«Рекомендую А, потому что проще. Б подойдёт если понадобится X» → удобно.

### ❌ Говорить «это сложно» без объяснения
Лучше: «Это займёт 3 сессии вместо одной, потому что [причина].
Хочешь продолжить?»

## Ошибки в архитектуре

### ❌ Строить на будущее вместо настоящего
MVP — это минимум. Не нужна микросервисная архитектура для 10 пользователей.

### ❌ Менять стек в середине проекта
«Давай перейдём с Supabase на Firebase» в середине MVP — это начать заново.

### ❌ Не думать о мобильных устройствах с самого начала
Переделать десктопный интерфейс под мобильный дороже, чем сделать
сразу адаптивным.

### ❌ Игнорировать права доступа на старте
Добавить авторизацию и роли в конце — значит переписать половину логики.

## Сигналы что что-то идёт не так

Агент должен остановиться и сообщить владельцу если:
- Задача не закрывается больше 3 сессий → предложить пересмотреть и разбить
- Исправление одного бага ломает три других
- Непонятно как протестировать сделанное
- Для следующего шага нужно переписать предыдущий

> **Эскалация по длительности задачи:**
> После 2-й сессии без закрытия задачи — написать владельцу: «Задача идёт дольше ожидаемого. Разбить на части или продолжить?»
> Это не признак неудачи — это норма для сложных задач.

> **Правило размера задачи:** если при планировании задача выглядит больше одной сессии — разбить сразу на подзадачи TASK-XXXa / TASK-XXXb. Не ждать пока «застрянем». Три сессии без закрытия — симптом, что задачу надо было разбить ещё на старте.

## ⚠️ Анти-паттерны AI-специфичные (скрытые риски)

### 🔇 Silent Failure Mode — тихий отказ
**Описание:** Агент генерирует код, который компилируется и проходит тесты,
но работает неверно: отдаёт неправильные данные, медленно, или в граничных случаях.
**Почему опасно:** всё «зелёное», но продукт ведёт себя неожиданно в реальном использовании.
Для медицинских систем это критично — неверный расчёт дозы или пропущенный статус пациента
могут не вызвать ошибки компиляции.
**Защита:** формулируй acceptance-критерии ДО генерации кода.
Добавляй тест с реальными данными, а не только с «идеальным» входом.
Пример критерия: `Успех = система отклоняет дозу > 2 мг/кг для пациентов младше 12 лет`.

### 📚 Epistemic Debt — долг непонимания
**Описание:** Ты принял результат агента, не понимая что именно он сделал и почему.
При следующей ошибке — ты не сможешь вмешаться осмысленно.
**Почему опасно:** агент несёт ответственность только за генерацию.
Ты — единственный гарант качества. Если ты не понял код — нет второго страхующего.
**Защита:** после каждого крупного блока кода — попроси агента объяснить
ключевые решения простым языком до того, как идти дальше.
Фраза: «Объясни что ты сделал в двух предложениях, как будто я не программист».

### ✅ Over-reliance on Tests — ложная безопасность тестов
**Описание:** Агент пишет тесты под код, который сам же написал.
Тесты проходят — но они проверяют реализацию, а не требование.
**Почему опасно:** возникает иллюзия покрытия. В реальности тест и баг
«договорились» ещё при генерации.
**Защита:** формулируй сценарии проверки ДО написания кода.
Скажи агенту: «Сначала напиши критерии — что должно работать, потом код».
Тест пишется по требованию, а не по реализации.
