# UX-CHECKLIST-MEDICAL.md

Медицинский UX-чек-лист для агента.  
Базовые UX-пункты (форма, навигация, empty states, общие ошибки) берутся из разделов **DEFAULT / MINIMAL** в [`ux-checklist-core.md`](./ux-checklist-core.md); здесь остаётся только клиническая специфика.

## ⚖️ Комплаенс 152-ФЗ

- [!] До сбора медицинских ПДн показывается согласие и ссылка на политику.
- [!] Пользователь видит цель обработки и объём собираемых данных в понятной формулировке.
- [!] В интерфейсе есть явный сценарий отзыва согласия и удаления/блокировки данных.
- [!] Критичные действия с ПДн подтверждаются и попадают в журнал.
- [!] Для реализации юридических текстов и форм используется [`LEGAL-152FZ.md`](./LEGAL-152FZ.md).

## 1. Клиническая безопасность интерфейса

- [!] Критичные клинические показатели всегда выше второстепенных данных.
- [!] Статусы риска и срочности передаются не только цветом, но и текстом.
- [!] Любое рискованное действие врача требует явного подтверждения.
- [!] Ошибки формулируются безопасно: что случилось и что делать дальше без двусмысленности.

## 2. Медицинский контекст пациента/случая

- [!] На экране всегда видны идентификатор случая, текущий статус и ответственный.
- [!] Единицы измерения и референсные интервалы отображаются рядом с показателями.
- [ ] Есть закреплённый клинический summary пациента/случая.
- [ ] Есть блок последних критичных событий по пациенту.

## 3. Трассируемость врачебных действий

- [!] Видно, кто и когда внёс клиническое решение или изменение назначения.
- [!] После критичного действия пользователь получает явное подтверждение результата.
- [ ] Доступен таймлайн изменений по пациенту/случаю.
- [ ] Доступен аудит действий по ролям.

## 4. ⚕️ Клинические сценарии

- [!] Валидация дозировок: значение, единицы и допустимый диапазон проверяются до сохранения.
- [!] Журналирование врачебных решений: фиксируется основание решения и ответственный.
- [!] Телемедицинское информированное согласие: согласие подтверждается до консультации и хранится с отметкой времени.
- [!] Приоритизация симптомов: красные флаги отображаются первыми и требуют отдельного подтверждения действия.
- [!] Доступность для пациентов с ОВЗ: крупный шрифт, контраст, понятные формулировки, поддержка экранных ассистентов.

## 5. Ролевой медицинский UX

- [!] Интерфейсы врача, пациента и администратора разделены по целям и доступам.
- [!] Пользователь не видит клинические данные, которые не нужны его роли.
- [ ] Для каждой роли есть отдельный сценарий первого входа.

## 6. Правило применения

- Сначала применять базовые разделы в [`ux-checklist-core.md`](./ux-checklist-core.md), затем этот блок как медицинское усиление.
- При конфликте «скорость vs клиническая безопасность» всегда приоритет у безопасности.
- Для данных пациента и телемедицины обязательно сверять решения с [`LEGAL-152FZ.md`](./LEGAL-152FZ.md).
