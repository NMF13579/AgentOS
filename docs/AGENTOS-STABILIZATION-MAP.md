# AgentOS Stabilization Map (M32.1)

## Purpose

M32.1 фиксирует карту рисков стабилизации до любых исправлений.
Цель: сделать видимыми места, где AgentOS может ошибаться, пропускать блокировки, давать ложный "OK" или вводить пользователя в заблуждение.

## Stabilization Principle

1. Сначала картировать режимы отказа (failure modes = как система может сломаться).
2. Затем проверить workflow (workflow = последовательность шагов работы).
3. Затем проверить зоны ответственности скриптов.
4. Затем проверить negative coverage (negative coverage = покрытие негативных сценариев, где должно быть BLOCK/FAIL).
5. Затем спроектировать audit (audit = проверка с доказательствами).
6. Затем приоритизировать исправления.
7. Реализация переносится в M33.

## Layer 1 — Technical Failure Modes

Цель слоя: AgentOS не должен выдавать ложный зелёный статус.

Ключевые технические риски:

- `UNKNOWN` становится `OK` из-за неверной интерпретации статуса.
- Устаревший report (report = отчёт проверки) трактуется как свежий.
- Повреждённый report не блокирует следующий шаг.
- Отсутствует Context Pack (контекстный набор), но выполнение продолжается.
- Поддельное approval (approval = подтверждение человека) принимается как настоящее.
- Небезопасный next step (следующий шаг) помечается как допустимый execute (execute = выполнение).
- Renderer (renderer = слой отображения) скрывает состояние `BLOCKED`.
- CLI wrapper (обёртка командной строки) начинает показывать команды `fix/go/approve/ready` как управляющие.
- Audit начинает менять файлы, хотя должен только проверять.
- Результат blocked-подсистемы интерпретируется как allowed.

Ориентиры проверки:

- Единая трактовка `OK/UNKNOWN/BLOCKED/FAIL` во всех слоях.
- Явная проверка свежести артефактов.
- Жёсткий запрет на «зелёный» результат при повреждённых/неполных входных данных.

## Layer 2 — Workflow Failure Modes

Ожидаемый путь:

`status` → `why` → `next` → `validate` → `audit` → `evidence` → `review` → `completion`

Цель слоя: пользователь и агент проходят через понятные gate (gate = обязательные контрольные шаги) без прыжков через этапы.

Ключевые риски:

- Неясна первая команда для старта.
- Слишком много разрозненных скриптов без единой траектории.
- Нет единого "check everything" пути.
- Этап review можно пропустить.
- Evidence (evidence = доказательства проверки) есть, но свежесть неочевидна.
- Completion review не проверяет результат audit.
- Workflow опирается на память человека вместо формальных блокировок.

Ориентиры проверки:

- Для каждого шага есть вход, выход и явное условие перехода.
- Нельзя перейти к completion без подтверждённого review + audit.
- Нельзя использовать устаревшие доказательства как основание завершения.

## Layer 3 — Script Responsibility / Drift

Принцип: one meaning — one owner (один смысл — один владелец ответственности).

| Script / Component | Owner Responsibility | Must Not Do | JSON? | Exit Codes? | Mutates Files? | Drift Risk |
|---|---|---|---|---|---|---|
| `scripts/agentos-status.py` | Только вычисление машинного статуса | Обучать/объяснять, принимать решения за workflow | Yes | Yes | No | Статус начинает "учить", а не сообщать |
| `scripts/agentos-view-model.py` | Только подготовка данных для отображения | Самостоятельно менять статус/правила | Yes | No | No | View-model начинает решать за status |
| `scripts/agentos-explain.py` | Только объяснение статуса человеку | Менять источники истины или статус | No | No | No | Объяснение подменяет факты |
| `scripts/agentos-next-step.py` | Только подсказка следующего шага | Выполнять действие или обходить блокировки | Yes | Yes | No | "Next" становится запуском действий |
| `renderer` | Только отображение входных данных | Читать отчёты напрямую и менять логику | No | No | No | Renderer скрывает/смягчает `BLOCKED` |
| `scripts/agentos.py` | Только роутинг (routing = маршрутизация команд) | Превращаться в control panel (панель управления) | No | Yes | No | Обёртка начинает давать команды управления |
| `audit` (`scripts/audit-agentos.py`) | Только проверка состояния и вывод результата | Исправлять, перезаписывать, авто-лечить | Yes | Yes | No | Audit получает `--fix` и теряет нейтральность |

Системные drift-риски:

- `status` начинает «тьюторить» вместо строгого результата.
- `view-model` начинает определять финальный статус.
- `renderer` читает отчёты напрямую, минуя нормализованные входы.
- CLI wrapper эволюционирует в управляющую панель.
- `audit` получает режим `--fix`.
- Разные скрипты по-разному трактуют `UNKNOWN`.

Цель слоя: исключить размывание ролей между скриптами и интерфейсом.

## Layer 4 — Human/Product Failure Modes

Цель слоя: AgentOS должен быть понятным человеку без создания ложного ощущения управления.

Ключевые риски:

- Пользователь не понимает, почему задача остановлена.
- Слово `next` воспринимается как немедленное выполнение.
- Checklist (чеклист) выглядит как approval.
- Dashboard (дашборд = сводный экран) выглядит как control panel.
- Цвета выглядят «официальным разрешением», даже если данных недостаточно.
- Human Gate (человеческий контрольный шаг) превращается в слепую галочку.
- Tutor/объяснение слишком техническое.
- `BLOCKED` выглядит как мягкое предупреждение, а не стоп.

Ориентиры проверки:

- Причина стопа объясняется простым языком.
- Текст не обещает выполнение там, где есть только подсказка.
- Визуальные сигналы не заменяют формальные правила допуска.

### Severity Definition

- `P0` — false OK / bypass / fake approval (ложный OK, обход, фейковое подтверждение).
- `P1` — workflow jump / stale evidence / unclear command (прыжок по этапам, устаревшие доказательства, неясная команда).
- `P2` — duplicated logic / weak docs / missing examples (дубли логики, слабая документация, нехватка примеров).
- `P3` — polish / future productization (полировка и отложенная продуктовая доработка).

## M32 Follow-up Map

### 32.2 Workflow Review

- Purpose: проверить путь `status → why → next → validate → audit → evidence → review → completion` как единый маршрут.
- Expected artifact: документ с картой обязательных переходов и точек, где нельзя прыгать этапы.
- Must not implement yet: новые команды, авто-роутинг, изменение runtime-логики.

### 32.3 Script Responsibility Review

- Purpose: зафиксировать жёсткие границы ответственности каждого скрипта/компонента.
- Expected artifact: матрица ответственности и анти-паттернов drift.
- Must not implement yet: рефакторинг скриптов, изменение exit-кодов, новые флаги.

### 32.4 Negative Fixture Coverage Map

- Purpose: определить, какие негативные сценарии уже покрыты и какие отсутствуют.
- Expected artifact: карта негативных фикстур и пробелов покрытия по статусам/гейтам.
- Must not implement yet: добавление новых тестов, валидаторов, фикстур в код.

### 32.5 Stabilization Audit Design

- Purpose: спроектировать детерминированный audit-проход без изменений рабочих файлов.
- Expected artifact: спецификация аудита (входы, проверки, критерии PASS/BLOCK/FAIL, доказательства).
- Must not implement yet: реализация audit-логики, режимы `--fix`, автовосстановление.

### 32.6 Stabilization Priority Plan

- Purpose: приоритизировать задачи стабилизации по уровню риска P0→P3.
- Expected artifact: план очередности исправлений и зависимостей.
- Must not implement yet: выполнение исправлений, изменения CI/валидаторов/runtime.

## Non-Goals

M32.1 явно **не** реализует:

- audit logic (логика аудита);
- validators (валидаторы);
- self-heal (самоисправление);
- recovery (автовосстановление);
- shadow branches;
- git checkpoints;
- packaging;
- TUI/web UI;
- new runtime enforcement (новые механизмы принудительного контроля во время выполнения);
- new product-facing CLI commands (новые пользовательские CLI-команды).

## Final Boundary

M32.1 — это карта, а не исправление.
Никакая реализация стабилизации не начинается, пока карта не существует.
