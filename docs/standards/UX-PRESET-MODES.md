# UX-PRESET-MODES.md

Режимы UX-настроек для проекта.
Пользователь не выбирает отдельные чек-листы вручную.
Агент сам подбирает подходящий режим и объясняет, что в него уже включено.

## 1. Общий принцип

Агент не начинает с вопроса:
“Какой чек-лист выбрать?”

Агент начинает с предложения:
“Я рекомендую такой UX-режим для этого проекта.”

Цель:
- сократить количество решений на старте;
- создать ощущение, что система уже позаботилась о базовом UX;
- оставить пользователю право упростить или расширить набор.

## 2. Доступные режимы

### 2.1. Базовый режим

Для большинства web-проектов, кабинетов и обычных интерфейсов.

Включает:
- `UX-CHECKLIST-DEFAULT.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`
- `UX-CHECKLIST-EMPTY-STATES.md`

Что уже покрыто:
- структура экранов;
- формы, ошибки, success states;
- пустые состояния;
- базовая доступность.

Когда рекомендовать:
- если проект не выглядит узкоспециализированным;
- если нет явного mobile-first или медицинского приоритета;
- если нужен “нормальный стандартный интерфейс без лишнего”.

### 2.2. Минимальный режим

Для MVP, простых сервисов, быстрых интерфейсов и случаев, когда важна простота.

Включает:
- `UX-CHECKLIST-MINIMAL.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`
- `UX-CHECKLIST-EMPTY-STATES.md`

Что уже покрыто:
- только базовые элементы интерфейса;
- понятные формы и ошибки;
- минимальная когнитивная нагрузка.

Когда рекомендовать:
- если пользователь просит “просто и без перегруза”;
- если проект на стадии MVP;
- если сначала нужен быстрый запуск, а не богатый UX.

### 2.3. Медицинский режим

Для медицинских и около-медицинских проектов.

Включает:
- `UX-CHECKLIST-MEDICAL.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`
- `UX-CHECKLIST-EMPTY-STATES.md`
- `UX-CHECKLIST-NOTIFICATIONS.md`
- `UX-CHECKLIST-HISTORY-AND-AUDIT.md`
- `UX-CHECKLIST-ROLE-BASED-UX.md`

Что уже покрыто:
- безопасные сценарии;
- подтверждение чувствительных действий;
- понятные статусы;
- история изменений;
- role-based интерфейсы;
- доступность и пустые состояния.

Когда рекомендовать:
- если есть врач, пациент, администратор;
- если интерфейс работает с чувствительными данными;
- если ошибка пользователя может быть критичной.

### 2.4. Mobile-first режим

Для мобильных интерфейсов и проектов, где мобильный сценарий важнее desktop.

Включает:
- `UX-CHECKLIST-MOBILE.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`
- `UX-CHECKLIST-EMPTY-STATES.md`

Что уже покрыто:
- touch-friendly интерфейс;
- упрощённая навигация;
- mobile-формы;
- базовые состояния;
- доступность на мобильных устройствах.

Когда рекомендовать:
- если проект используют в дороге;
- если основной сценарий идёт со смартфона;
- если нужен mobile-first интерфейс.

### 2.5. Data-heavy режим

Для систем со списками, таблицами, журналами, реестрами и большими выборками данных.

Включает:
- `UX-CHECKLIST-DEFAULT.md`
- `UX-CHECKLIST-SEARCH-FILTERS.md`
- `UX-CHECKLIST-TABLES-AND-BULK-ACTIONS.md`
- `UX-CHECKLIST-EMPTY-STATES.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`

Что уже покрыто:
- поиск;
- фильтры;
- сортировка;
- таблицы;
- массовые действия;
- состояния “ничего не найдено”.

Когда рекомендовать:
- если в проекте много записей;
- если пользователь работает с реестрами и списками;
- если без фильтров и поиска интерфейс станет неудобным.

### 2.6. Dashboard режим

Для обзорных экранов, KPI, очередей и контрольных панелей.

Включает:
- `UX-CHECKLIST-DASHBOARDS.md`
- `UX-CHECKLIST-NOTIFICATIONS.md`
- `UX-CHECKLIST-ACCESSIBILITY.md`
- `UX-CHECKLIST-EMPTY-STATES.md`

Что уже покрыто:
- KPI и статусы;
- приоритетные блоки;
- алерты и уведомления;
- обзорные экраны;
- пустые состояния.

Когда рекомендовать:
- если нужен dashboard;
- если продукт должен быстро показывать “что сейчас важно”;
- если есть очереди, статусы, приоритеты и события.

## 3. Правило выбора режима агентом

Агент сам подбирает режим по типу проекта.

### 3.1. Если проект обычный
Выбрать:
- Базовый режим.

### 3.2. Если проект очень простой или пользователь просит “без перегруза”
Выбрать:
- Минимальный режим.

### 3.3. Если проект медицинский или около-медицинский
Выбрать:
- Медицинский режим.

### 3.4. Если главный сценарий мобильный
Добавить или выбрать:
- Mobile-first режим.

### 3.5. Если проект построен вокруг реестров, таблиц и поиска
Добавить:
- Data-heavy режим.

### 3.6. Если ключевой экран — обзорная панель
Добавить:
- Dashboard режим.

## 4. Комбинирование режимов

Агент может комбинировать режимы.

Примеры:
- Медицинский + Mobile-first.
- Медицинский + Dashboard.
- Базовый + Data-heavy.
- Минимальный + Mobile-first.

Если режимов несколько, агент должен:
- выбрать один как основной;
- остальные обозначить как дополнительные расширения.

## 5. Как агент объясняет выбор

Агент не перечисляет пользователю длинный список файлов.
Агент объясняет выбор человеческим языком.

Пример:
“Для этого проекта рекомендую медицинский режим с mobile-first расширением.
По умолчанию туда уже входят безопасные сценарии, понятные статусы, история изменений, уведомления, пустые состояния, доступность и мобильные паттерны.
Это покроет основные UX-риски сразу.
Если хотите, можем упростить до минимального режима.”

## 6. Если пользователь не хочет выбирать

Если пользователь не хочет обсуждать режимы подробно:
- агент сам включает наиболее подходящий режим;
- кратко объясняет, что уже включено;
- даёт возможность отключить отдельные блоки позже.

## 7. Главный принцип

Пользователь выбирает не чек-листы, а понятный режим.
Чек-листы остаются внутренней системой сборки UX-стандарта для агента.
Это делает старт беседы проще, быстрее и интуитивнее.
