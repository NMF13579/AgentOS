# UX-CHECKLIST-ACCESSIBILITY.md

Чек-лист доступности (a11y) для агента.
Применяется ко всем интерфейсам, особенно для медицинских и мобильных.
`[x]` — включено по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Общие принципы

- [!] Основные сценарии должны быть доступны без мыши.
- [!] Смысл не должен передаваться только цветом.
- [!] Текст должен быть читаем без зума и танцев с бубном.
- [x] Критичные действия и статусы должны быть понятны любому пользователю.
- [ ] Отдельный обзор доступности перед релизом.

## 2. Текст и контраст

- [!] Контраст текста и фона соответствует WCAG AA.
- [!] Основной текст не мельче базового размера (обычно 16 px).
- [x] Второстепенный текст остаётся читаемым.
- [x] Текст на цветных кнопках/бейджах читабелен.
- [x] Нет серого текста на слегка более сером фоне.
- [ ] Проверка контраста для всех ключевых элементов отдельно.

## 3. Цвет и смысл

- [!] Цвет не единственный способ обозначить статус.
- [!] Ошибки и предупреждения имеют и текст, и цвет.
- [x] Важные статусы дополнительно маркируются иконкой или текстом.
- [x] Легенды к цветам в графиках и таблицах понятны.
- [ ] Добавлены текстовые метки к цветовым индикаторам.

## 4. Клавиатурная навигация

- [!] Все основные действия доступны с клавиатуры.
- [!] Фокус по Tab двигается логично и предсказуемо.
- [!] Фокус видно (подсветка, рамка, смена состояния).
- [x] Кнопки и ссылки можно активировать Enter/Space.
- [ ] Есть "перейти к основному контенту" (skip link).
- [ ] Комбинации клавиш для частых действий (где уместно).

## 5. Screen reader и семантика

- [!] Заголовки выстроены по уровню (h1, h2, h3 без хаоса).
- [!] Основные блоки размечены семантически (header, main, nav, footer).
- [!] Интерактивные элементы имеют корректные роли (button, link и т.п.).
- [x] У иконок‑кнопок есть понятный текст (aria-label).
- [x] У картинок есть alt или явное пустое alt, если картинка декоративная.
- [ ] ARIA-атрибуты используются только там, где действительно нужны.

## 6. Формы и ошибки

- [!] Каждое поле имеет подпись (label).
- [!] Ошибка привязана к конкретному полю.
- [!] Ошибки объясняют, что сделать, а не только “неверное значение”.
- [x] Обязательные поля явно отмечены.
- [x] Сообщения об успехе и ошибке понятны на слух.
- [ ] Есть общее summary ошибок для длинных форм.

## 7. Фокус, модальные окна, диалоги

- [!] При открытии модального окна фокус переходит внутрь него.
- [!] При закрытии модального окна фокус возвращается туда, откуда пришёл.
- [!] В модальных окнах есть явная кнопка закрытия.
- [x] Escape закрывает диалог, если это безопасно.
- [x] Пользователь не может “потерять” фокус в невидимой области.
- [ ] Тёмная подложка под модальным окном не мешает работе screen reader.

## 8. Мобильная доступность

- [!] Интерактивные элементы не менее 44x44 px.
- [!] Расстояние между touch‑целями достаточное.
- [!] Основные действия доступны без точного прицеливания.
- [x] Интерфейс корректно работает в портретной ориентации.
- [ ] Поддерживается landscape, если сценарий этого требует.
- [ ] Учитываются системные жесты (без конфликтов).

## 9. Состояния элементов

- [!] Есть состояния: обычное, наведение (если есть), фокус, нажатие, отключено.
- [!] Состояния понятны не только по цвету (форма, иконка, текст).
- [x] Disabled выглядит реально недоступным, но читабельным.
- [x] Загрузка отображается явно (спиннер, skeleton, текст).
- [ ] Важные долгие операции имеют индикатор прогресса.

## 10. Контент и язык

- [!] Формулировки максимально ясные, без лишнего жаргона.
- [!] Длинные фразы разбиты на удобочитаемые блоки.
- [x] Сокращения и аббревиатуры объясняются там, где это критично.
- [x] Важное не прячется в “мелкий серый текст”.
- [ ] Есть глоссарий, если много спецтерминов.

## 11. Медицинский контекст

- [!] Сообщения не вызывают лишней тревоги, если ситуация не критична.
- [!] Важные статусы (например, критические результаты) обозначены ясно.
- [!] Ошибки не порождают ощущение “данные пациента потеряны”.
- [x] Подробные технические детали скрыты за дополнительным действием.
- [ ] Разные роли (врач, пациент, администратор) получают свой уровень детализации.

## 12. Проверка и поддержка

- [x] Есть базовый чек‑лист, по которому агент проходит перед релизом.
- [x] Основные сценарии проверены с клавиатурой.
- [x] Основные сценарии проверены на мобильном устройстве.
- [ ] Проведено отдельное тестирование доступности для медицинских интерфейсов.
- [ ] Запланирован регулярный пересмотр доступности по мере развития продукта.

## 13. Правило применения

- Агент использует все пункты `[!]` как обязательные для любого интерфейса.
- Пункты `[x]` считаются стандартом доступности по умолчанию.
- Пункты `[ ]` агент предлагает как дополнительные меры для повышенной доступности.
- В случае конфликта между визуальным “вау” и доступностью агент выбирает доступность и понятность.

## 14. Фраза для согласования с пользователем

"Могу включить стандартный чек‑лист доступности: контрастный текст, понятные заголовки, работа с клавиатуры, корректные сообщения об ошибках, доступные формы и мобильные touch‑элементы. Обязательные пункты будут учтены во всех экранах. Применяем?"
