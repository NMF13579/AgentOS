# UX-CHECKLIST-NOTIFICATIONS.md

Чек-лист уведомлений для агента.
Цель — не потерять важные события и не утопить пользователя в шуме.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Типы уведомлений

- [x] Уведомления об успехе действия (success).
- [x] Уведомления об ошибке (error).
- [x] Предупреждения (warning).
- [x] Информационные уведомления (info).
- [ ] Напоминания (reminder).
- [ ] Системные уведомления (обновления, регламент).

## 2. Каналы уведомлений

- [!] Inline-уведомления в контексте экрана.
- [x] Временные всплывающие уведомления (toast).
- [ ] Панель уведомлений/инбокс.
- [ ] E-mail / внешние каналы (фиксируются отдельно).

## 3. Правила приоритета

- [!] Критичные уведомления не должны теряться среди неважных.
- [x] Критичные события показываются ближе к центру внимания (верх экрана, центр).
- [x] Второстепенные события могут быть только в виде toast или в панели уведомлений.
- [ ] Возможность просмотреть историю важных уведомлений.

## 4. Success-уведомления

- [x] Понятно написано, что именно успешно сделали.
- [x] Нет лишнего шума (кратко, по делу).
- [x] Минимальный визуальный акцент (зелёный/успешный статус).
- [ ] Ссылка "посмотреть результат" или "перейти к записи".

## 5. Error-уведомления

- [!] Объясняют, что произошло, простыми словами.
- [!] Подсказывают, что делать дальше.
- [!] Понятно, где именно ошибка (форма, сеть, доступ).
- [x] Для ошибок формы — сообщение возле поля.
- [x] Для системных ошибок — понятное общее сообщение.
- [ ] Код ошибки/детали — отдельно, для поддержки.

## 6. Warning-уведомления

- [x] Обозначают потенциальную проблему, но не блокируют работу.
- [x] Дают возможность пользователю осознанно продолжить или изменить действие.
- [ ] Можно скрыть предупреждение, если оно не критично.
- [ ] Отдельная визуальная маркировка (жёлтый/оранжевый).

## 7. Info-уведомления

- [x] Используются редко и по делу.
- [x] Не маскируют критичные уведомления.
- [ ] Содержат короткий совет или факт.
- [ ] Возможность скрыть и не показывать снова.

## 8. Напоминания

- [ ] Есть явный список событий, по которым допустимы напоминания.
- [ ] Напоминания не мешают критичным действиям.
- [ ] Пользователь может отключить или настраивать часть напоминаний.
- [ ] Для медицинских сценариев напоминания сформулированы максимально нейтрально и точно.

## 9. Визуальная иерархия

- [x] Критичные уведомления визуально заметнее.
- [x] Цвета и иконки согласованы между экранами.
- [x] Текст не мельче основного текста в интерфейсе.
- [ ] Есть единый стиль для всех уведомлений в продукте.

## 10. Медицинский контекст

- [!] Формулировки не вызывают лишней тревоги при некритичных событиях.
- [!] Критичные медицинские события выделены явно.
- [!] Уведомления не создают ощущение “данные потеряны”, если это не так.
- [ ] Для пациента и врача используются разные формулировки, соответствующие роли.
- [ ] Важные уведомления можно найти повторно (история).

## 11. Правило применения

- Агент использует `[!]` как обязательные требования.
- Агент применяет `[x]` как стандарт по умолчанию.
- `[ ]` предлагает как дополнительные опции под сценарий.
