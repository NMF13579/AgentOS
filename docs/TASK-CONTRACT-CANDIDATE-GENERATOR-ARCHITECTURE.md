# Task Contract Candidate Generator Architecture

## Purpose

Define M51 architecture for safe generation of M50-compatible artifacts without introducing new lifecycle authority.
The architecture defines how a controlled generator produces conversion-package artifacts compatible with M50 validation, while preserving non-execution boundaries.

## M51 Boundary Summary

M51 documents generator architecture only.
M51 does not create generator behavior in this task.
M51 does not create generated artifacts in this task.
M51 does not place anything into queue and does not modify active task state.
M51 does not authorize execution, placement, or implementation.

## Relationship to M50

M51 depends on M50 as the authority layer for conversion-package shape and candidate boundary semantics.
Generated candidate semantics must stay compatible with M50 EXECUTION_SHAPE.
M51 must not bypass M50 validator compatibility requirements.

## Core Generator Flow

validated conversion package
↓
controlled generator
↓
dry-run generated conversion package preview
↓
optional written generated conversion package in safe staging area
↓
M50 validator check
↓
generated candidate remains ready for M52/M53 review

M51 stops after generation and validation compatibility.

## Definitions

- controlled generator: bounded component that transforms validated conversion input into M50-compatible generated conversion package output.
- validated conversion package: input package that already satisfies M50 conversion input and boundary requirements.
- generated conversion package: produced package artifact containing conversion_package plus embedded candidate content.
- embedded task_contract_candidate: candidate content nested inside the generated conversion package and kept in EXECUTION_SHAPE-compatible form.
- primary generated artifact: the main file generated by M51 for validator compatibility checks.
- primary validator target: artifact path passed to the M50 validator command.
- safe staging area: controlled path family where generated artifacts may be written in later M51 tasks.
- dry-run: non-writing preview mode that predicts output location and content intent.
- write mode: mode that actually writes generated artifact to the safe staging area.
- would_write_to: predicted path emitted by dry-run.
- output_path: actual file path written in write mode.
- primary_validator_target: concrete path of generated artifact used for validator invocation.
- queue entry: lifecycle object placed in tasks/queue/ for later selection.
- active task state: lifecycle state represented by tasks/active-task.md.
- execution authorization: permission to run implementation work.
- placement gate: separate lifecycle decision layer that decides queue/active placement.

## Required Inputs

- validated conversion package
- source traceability references required by M50
- scoped conversion authorization reference required by M50
- carry-forward fields required by M50 (limitations, open questions, downstream limits, non-authority boundary)

## Allowed Outputs

- dry-run preview metadata (non-writing)
- optional written generated conversion package in safe staging area (later M51 tasks only)
- M50 validator-compatible primary artifact target reference

## Primary Output Format

generated/task-contract-candidates/<candidate-id>.generated-conversion-package.md

Primary output:
- generated conversion package with embedded task_contract_candidate

## M50 Validator Compatibility Requirement

Generated output must be compatible with:

python3 scripts/validate-proposal-to-task-conversion.py --conversion <path>

M51 generated output must preserve the M50 conversion package shape expected by the M50 validator.

M51 must not generate a standalone candidate-only artifact as the primary validator target if M50 validator requires conversion_package + embedded task_contract_candidate.

## Dry-Run First Architecture

Dry-run is the default safety-first path.
Dry-run provides deterministic preview metadata and planned pathing only.
Dry-run does not create files and does not mutate lifecycle state.

## Write-to-Staging Architecture

Write mode is optional and only for explicitly permitted later M51 tasks.
Write mode stores generated artifact in safe staging area and sets validator target to that written path.
Write mode still does not authorize execution, placement, or implementation.

## Safe Staging Area

M51 may write generated artifacts only inside generated/task-contract-candidates/.
M51 must not write to tasks/active-task.md.
M51 must not write to tasks/queue/.
M51 must not write to approvals/.
M51 must not create approval records.

This allowance applies only to later M51 tasks that explicitly permit generated artifacts.
Task 51.1 does not create generated/.
Task 51.1 only documents the generated/task-contract-candidates/ staging boundary.

## Dry-Run and Write JSON Semantics

dry-run predicts would_write_to
write creates output_path
only write creates primary_validator_target

In dry-run mode:
- written must be false
- output_path must be null
- primary_validator_target must be null
- would_write_to must contain the predicted staging path that would be written if --write were used

In write mode:
- written must be true
- output_path must contain the actual generated artifact path
- primary_validator_target must equal the actual generated artifact path
- would_write_to may equal output_path or be omitted if output_path is present

Dry-run predicted output path is not a written artifact.
Dry-run does not create lifecycle state.
Write to staging does not create lifecycle state.

## Generated Candidate Boundary

Generated candidate is not active task state.

Generator output is not implementation approval.

Generator must not write to tasks/queue/.
Generator must not create approval records.
Generator PASS is not approval.
Generator dry-run is not lifecycle transition.

## Generator Non-Authority Rules

Task contract candidate generator creates M50-compatible generated conversion packages only.
Task contract candidate generator does not authorize execution.
Task contract candidate generator does not place candidates into queue.
Task contract candidate generator does not modify active task state.
Task contract candidate generator does not create approval records.

Generated output still requires M52 validation or review.
Generated output still requires M53 placement gate before queue or active state.
M51 completion does not authorize M52 implementation.
M51 completion does not authorize M53 placement.

## Examples Boundary

M51 may create files under examples/ only for generator input demonstration.

This allowance applies to later M51 tasks that explicitly permit examples/.

Task 51.1 does not create examples/.

Task 51.1 only documents the examples/ boundary for later M51 tasks.

examples/ artifacts are documentation and test/demo inputs.
examples/ artifacts are not lifecycle state.
examples/ artifacts must not be copied into tasks/active-task.md.
examples/ artifacts must not be placed into tasks/queue/.
examples/ artifacts do not authorize execution, queue placement, active-task replacement, or implementation.

## Downstream Handoff to M52 and M53

M52 may validate generated task contract candidate quality, but M52 must not place candidates into queue.
M53 may define controlled queue / active placement as a separate lifecycle decision layer.
M51 does not grant M52 or M53 implementation authority.
