# Execution Result Verification Preconditions Contract

## Purpose

This contract defines the preconditions required to verify the execution result of a task in milestone M59. It outlines the checks to run, the structures to validate, and the outputs of the preconditions checking process.

## Preconditions

All task preconditions have been successfully verified:
- `reports/m59-m58-completion-intake.md` exists and contains `MAY_PROCEED_TO_M59_VERIFICATION_PLANNING: true`.
- `docs/EXECUTION-RESULT-VERIFICATION-ARCHITECTURE.md` exists and contains `FINAL_STATUS: M59_ARCHITECTURE_DEFINED` and maps tasks to 59.14.
- `docs/EXECUTION-RESULT-VERIFICATION-INPUT-CONTRACT.md` exists and contains `FINAL_STATUS: M59_EXECUTION_RESULT_VERIFICATION_INPUT_CONTRACT_DEFINED`.
- JSON schema and templates for verification input exist and are correct.
- `reports/m58-completion-review.md` exists.
- The approved task map has not been expanded beyond 59.14.

## Position in M59

Preconditions (59.3) consume the verification input artifact (59.2). Preconditions evaluate whether all files, references, metadata, and constraints are structurally satisfied before passing control to git diff/scope verification (59.4) and validation evidence verification (59.5).

## Preconditions Artifact Role

The preconditions artifact is a structured report generated by the validation engine during dry-runs or pre-verification checks. It documents the evaluation of preconditions and decides whether to block, request information, or proceed with verification checks.

## Field Semantics

- **`preconditions_id`**: A unique string identifying the preconditions result artifact (e.g. `execution-result-verification-preconditions-<task_id>-<timestamp>`).
- **`task_id`**: Identifies the target task. Must match the value in the input.
- **`verification_input_id`**: The ID of the input being analyzed.
- **`preconditions_status`**: Represents readiness outcome.
- **`checked_at`**: Check execution timestamp.
- **`checked_by`**: Agent descriptor (metadata).
- **`source_references`**: Collection of exact artifact paths checked.
- **`m58_completion_status`**: Observed state of the M58 completion review.
- **`m59_intake_status`**: Observed state of the M59 completion intake.
- **`m59_architecture_status`**: Status of architecture configuration.
- **`input_contract_status`**: Status of input contract definition.
- **`input_status`**: State of the processed input submission.
- **`input_validity`**: Conformance score of the input file.
- **`execution_session_record_check`**: Results from session record file validation.
- **`declared_result_check`**: Declared completion statement status.
- **`declared_changes_check`**: Status of declared list of modifications.
- **`actual_changed_files_check`**: Status of physical git file list availability.
- **`git_diff_evidence_check`**: Status of git diff file availability.
- **`scope_input_check`**: Status of allowed paths and boundary files.
- **`validation_evidence_check`**: Presence and freshness of validation files.
- **`test_evidence_check`**: Presence and freshness of test runner outputs.
- **`forbidden_change_rule_checks`**: Compliance outcome for workspace safety markers.
- **`human_review_handoff_check`**: Verification of review request status.
- **`forbidden_action_checks`**: Security check outcomes for illegal commands.
- **`premature_downstream_artifact_checks`**: Checks for early file creations.
- **`warnings`**: Minor non-blocking concerns.
- **`blockers`**: Major blocking violations.
- **`handoff_to_git_diff_scope_verification`**: Boolean indicator to proceed to 59.4.
- **`handoff_to_validation_evidence_verification`**: Boolean indicator to proceed to 59.5.
- **`non_authority_acknowledgement`**: Acknowledges structural limitations.

## Allowed Preconditions Statuses

The `preconditions_status` must be one of:
- `EXECUTION_RESULT_PRECONDITIONS_READY`: Clean state, allows handoffs to proceed.
- `EXECUTION_RESULT_PRECONDITIONS_READY_WITH_WARNINGS`: Non-blocking concerns present, but allows handoffs to proceed with warnings preserved.
- `EXECUTION_RESULT_PRECONDITIONS_NOT_READY`: Key files missing or incomplete. Handoffs are blocked.
- `EXECUTION_RESULT_PRECONDITIONS_BLOCKED`: Violations, safety concerns, or contradictory claims found. Handoffs are blocked.

## Source Artifact Checks

Source checks ensure that `reports/m58-completion-review.md`, `reports/m59-m58-completion-intake.md`, `docs/EXECUTION-RESULT-VERIFICATION-ARCHITECTURE.md`, and the contract definitions exist, are valid, and contain required status keywords. Any failure blocks verification.

## Verification Input Validity Checks

Analyzes the target verification input artifact against the schema `schemas/execution-result-verification-input.schema.json`.
- Must resolve to `VERIFICATION_INPUT_VALID` to proceed.
- Schema violations, malformed layout, or missing required fields block.

## Execution Session Record Checks

Preconditions check that the referenced execution session record file exists.
- The record's status must be `EXECUTION_SESSION_RECORD_CLOSED_PENDING_M59_VERIFICATION` to proceed.
- `EXECUTION_SESSION_RECORD_READY_FOR_POLICY` may proceed with warning.
- `EXECUTION_SESSION_RECORD_BLOCKED`, `EXECUTION_SESSION_RECORD_ABORTED`, `UNKNOWN`, or `MISSING` block.

## Declared Result Checks

Verifies that the developer agent declared a summary, limitations, and unresolved items.
- Claiming complete (`claimed_complete: true`) is treated as a claim only, not validation proof.
- Claims of task completion approvals or verified results inside declarations block.

## Declared Changes Checks

Verifies that lists of files created, deleted, and changed are declared.
- Missing declared changes block or fail preconditions to `NOT_READY`.
- Declared changes are recorded as inputs for physical git validation in 59.4.

## Actual Changed Files Checks

Checks if the list of actual changed files is declared or available for discovery.
- If source is `git_diff`, the engine checks that the workspace can run read-only git queries.
- If source is `not_available`, preconditions fail to `NOT_READY`.
- Setting a source without matching files blocks.

## Git Diff Evidence Checks

Checks if a physical git diff artifact path is referenced or can be dynamically queried.
- Stale, unreadable, or missing diff evidence results in `NOT_READY` or `BLOCKED`.
- Git diff evidence is treated as source data, never as approval.

## Scope Input Checks

Checks that expected scope metadata (allowed files/directories, forbidden paths, protected paths) exists.
- `scope_expansion_allowed` must be false. Setting it to true blocks preconditions unless manually overridden.
- Missing allowed definitions blocks.

## Validation Evidence Checks

Checks if validation evidence files and exit codes are referenced.
- A claim that validation ran (`claimed_validation_run: true`) must match physical validation logs.
- Missing evidence blocks or yields `NOT_READY`.

## Test Evidence Checks

Checks if test commands, exit codes, and summary logs are referenced.
- Stale logs or missing test evidence results in `NOT_READY` or `BLOCKED`.
- Running wrong or unrelated tests blocks clean verification.

## Forbidden Change Rule Checks

Checks that the developer agent acknowledged all safety constraints:
- `no_approval_artifacts`, `no_lifecycle_mutation`, `no_merge_push_release`, `no_human_review_replacement`, `no_m60_cleanup_artifacts` must all be true.
- Any disabled rules block preconditions.

## Human Review Handoff Checks

Checks that `human_review_handoff_required` is true and no claims are made replacing human gates. Any replacement claim blocks.

## Forbidden Action Checks

Preconditions evaluate the workspace state to confirm no forbidden operations were performed:
- `result_already_verified` must be false.
- `task_marked_complete` must be false.
- `approval_record_created` must be false.
- `lifecycle_mutation_performed` must be false.
- `commit_created`, `push_performed`, `merge_performed`, `release_performed` must be false.
- `m60_cleanup_started` must be false.
- Any true value blocks preconditions.

## Premature Downstream Artifact Checks

Preconditions check that no M59 files from tasks 59.4 through 59.14 (excluding current 59.0-59.3) or M60 cleanup files exist. If found, it blocks verification.

## Warning and Blocker Semantics

- **Warnings**: Do not prevent progress but require user notification. Recorded in `warnings`.
- **Blockers**: Immediately result in `EXECUTION_RESULT_PRECONDITIONS_BLOCKED` and set handoffs to false. Recorded in `blockers`.

## Handoff to 59.4 Git Diff and Scope Verification

- `handoff_to_git_diff_scope_verification` is true if preconditions status is `READY` or `READY_WITH_WARNINGS`.
- Otherwise, it is false.

## Handoff to 59.5 Validation Evidence Verification

- `handoff_to_validation_evidence_verification` is true if preconditions status is `READY` or `READY_WITH_WARNINGS`.
- Otherwise, it is false.

## Non-Authority Rules

Every preconditions artifact must acknowledge the following non-authority statements:
- `non_authority_acknowledgement` must contain the exact non-authority text defined in this contract.

## Malformed Preconditions Conditions

An artifact is malformed if required schema fields are missing, invalid JSON structures exist, or enum checks fail.

## Unsafe Preconditions Conditions

Preconditions are unsafe and blocked if:
- M58 completion or M59 intake statuses are missing or blocked.
- Verification input asserts task approval or replaces human reviews.
- Git actions (push/merge/release) are authorized.
- Performed action audits show modifications to protected paths or M60 file creation.

## Relationship to 59.4 Git Diff and Scope Verification Contract

Once preconditions pass, the engine hands off control to 59.4 to verify whether physical file modifications match allowed scopes.

## Relationship to 59.5 Validation Evidence Contract

Once preconditions pass, the engine hands off control to 59.5 to verify whether the test log timestamps match the current git state and that test outcomes are positive.

## Relationship to 59.7 Policy

Policy consumes the precondition, git diff scope, and validation evidence result artifacts to determine the final milestone decision.
